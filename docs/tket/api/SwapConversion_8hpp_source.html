24 /*
25 NOTE on ENCODING: with 6 vertices, there are 15 possible edges or swaps.
26 Thus, we can encode a single swap by a number in the range 0-15 (using 0 to
27 denote "no swap").
28 
29 This fits into 4 bits exactly.
30 
31 Thus, a single 64-bit unsigned int can store any swap sequence of length <= 16.
32 We also have the added benefit that ints written in hexadecimal are easier for a
33 human to read, since each hex digit 0-9 or A-F corresponds to a single swap.
34 
35 An obvious optimisation is that adjacent swaps should be different;
36 and also, blocks of four zeros cannot occur within the encoding.
37 However, this would still only reduce the total number to about 30%,
38 so we'd still need 62 or 63 bits to represent all sequences of length <= 16.
39 So it's not worth trying fancy encodings to store more possible sequences in
40 fewer bits, without a good theoretical breakthrough to come up with a really
41 good way to encode and search through only optimal or "near optimal" sequences.
42 
43 If we desire in future to increase the number of vertices, we'd have to use at
44 least 5 bits per swap, so could only fit sequences of length <= 12 in a 64-bit
45 int. Of course, (8*7)/2 = 28 < 31, so we could store swaps on <= 8 vertices
46 instead of 6.
47 
48 */
49 
50 // Generally no checks on the input values, it's assumed that the caller
51 // knows how the table encoding works.
52 // The possible swaps (01), (02), (03), ..., (45) on vertices {0,1,2,3,4,5}
53 // are listed in a global vector, so with values 0,1,...,14.
54 // Adding 1 to the index gives possible values 1,2,...,15 for the swaps,
55 // and 0 means no swap. Thus a sequence of swaps is encoded by storing the bits
56 // in a uint, with first swap at the least significant bits, and so on with
57 // leftward shifts by 4 bits each time.
